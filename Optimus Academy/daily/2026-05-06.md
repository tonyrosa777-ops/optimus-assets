#### Chapter 1-9 — Introduction, Hermes basics, Paperclip first demo

Julian frames the course around the thesis that the Hermes-plus-OpenClaw combination "just changed the AI agent game forever" and lets a user run "24/7 AI employees that get smarter every single day. No coding, no big budget, just pure automation magic." The viewer is promised that by end-of-video they will "have your own AI team running." The framing leans into the employee-replacement metaphor ("they don't sleep, they don't complain, they just get stuff done") plus the self-learning angle ("the more you use them, the smarter they get").

**What Hermes is at a definition level.** Hermes is "a new AI agent" built by a team called Noose Research. The differentiator Julian repeats: it "does not forget anything" and "actually gets smarter every single day on its own. No retraining, no prompting." The user-model contrast is explicit. Most AI agents today follow the pattern: give it a task, it does the task, it forgets everything, the next day it is "like talking to a stranger." Hermes inverts that: a thing told on Monday is still known on Friday. It "builds a profile of you, your projects, your preferences, how you like things done." The framing is "an assistant that actually pays attention" rather than a chatbot that resets per session.

**The self-improving-skill mechanic (the headline claim).** The standout mechanic Julian focuses on is autonomous skill creation. Hermes "does not just remember things. It learns new skills by itself." Stated literally: "It creates its own skills." The flow Julian describes: a user asks Hermes to do a task. Hermes does it. Then Hermes "writes down how it did it as a reusable skill." Next time the user asks for something similar, Hermes does it "faster and better. No one told it to do that. It just does it on its own." This is the self-improving loop the rest of the course will build on. Important Optimus interpretation lens: this is the same pattern Anthropic describes as Claude Code "Skills" (procedural memory), which means Hermes has likely implemented a standard skill-discovery loop on top of an LLM with persistent storage, not invented a fundamentally new learning algorithm. Julian does not make that distinction in this chapter.

**Background, scheduled, and modular execution model.** Julian contrasts "reactive" AI tools (sit there waiting for a user to type) with Hermes, which "runs in the background like a real employee." Three concrete capabilities are named. First, scheduled tasks: set Hermes up like an alarm, but instead of waking the user, it "goes and does work for you." Use cases listed include running reports, checking things, automating workflows in parallel with the user's other work. Second, multi-platform reach: Hermes runs on "Telegrams, Slack, Discord, WhatsApp, even your command line." His framing: "It is not locked into one app. It goes where you go." The implication is messaging-channel native behavior, the same surface area OpenClaw is known for. Third, modular brain: "You can swap out the brain. You are not locked into one AI model. You can use different large language models depending on what you need." Plus deployment optionality: "run the whole thing locally on your own computer or you can run it in the cloud. Your choice."

**Hermes vs OpenClaw, the honest first comparison.** Julian states the answer to the "is Hermes the new OpenClaw" question directly: "they are not the same thing. They are built for different purposes." OpenClaw is "a control center. It connects all your AI tools together. It orchestrates everything. It is great at managing systems and routing tasks to the right place." Hermes "is about intelligence. It is about an agent that gets smarter over time." The toolbox metaphor: "OpenClaw is like giving someone a toolbox. Hermes is like giving someone a toolbox and then they figure out new ways to use those tools that you never even thought of." The point-by-point comparison Julian gives at this stage of the course: OpenClaw uses predefined skills (someone builds them, the agent uses them); Hermes creates its own skills automatically. OpenClaw is focused on automation and orchestration; Hermes is focused on evolving intelligence. Both have persistent memory, "but the way they approach the whole agent thing is fundamentally different." His verdict: "Is Hermes the new OpenClaw? My honest answer is no. It is not replacing OpenClaw. It is something different. It is a different philosophy. OpenClaw is about orchestrating systems. Hermes is about growing intelligence. And honestly, you might end up using both for different things." This squares cleanly with the existing [[../concepts/openclaw-multi-agent-orchestration]] vault note. Note that by chapter 54 Julian's framing has hardened into "Honestly, I'm over OpenClaw"; the course was stitched from segments recorded weeks apart and his sentiment shifts visibly across chunks.

**Four real use cases for Hermes.** Julian enumerates four concrete use cases. First, an automated research assistant set up to "go out and research topics for you every single day." Because it remembers what it has already found, "it does not repeat itself. It builds on what it already knows." Julian flags this as "huge for anyone creating content or running a business." Second, daily reports: Hermes generates reports on "performance data, industry news, competitor analysis" on a schedule. "You wake up and it is already done. No prompting needed." Third, a personal AI assistant: the big one. Because Hermes remembers preferences and projects, "it actually feels like a real assistant, not a chatbot that forgets you exist every time you close the tab." Fourth, workflow automation: chain tasks together. "One thing triggers another. And because it learns, it finds better ways to do those tasks over time. That is the self-improving loop."

**Honest limitations Julian names up front.** "I always keep it honest on this channel. Hermes is still early stage. It is not perfect." The named gotchas: the self-learning feature "can sometimes overwrite behaviors you actually liked. So you have to keep an eye on it." This is a real risk for production deployments; skill regression with no diff is the classic symptom. Hermes also has fewer integrations than "more established tools, but that is changing fast. The community around it is growing like crazy. People are building new integrations every week." And the learning is "not always spot-on. Sometimes it learns the wrong lesson from a task. That is just the reality of where we are with self-improving AI." The big-picture framing Julian closes the Hermes intro with: "We are moving from AI tools that you have to prompt every single time to AI agents that evolve on their own." He calls Hermes "one of the first agents that really shows what that future looks like."

#### Chapter 9-11 — How Paperclip works, monitoring & budgets, Multica intro

**Why orchestration matters.** Most people use AI one task at a time: ask ChatGPT a question, get an answer, done. Paperclip's pitch is that if you want AI to actually run your business rather than just answer questions, you need it to write emails, post content, handle support, build products, and report back, all simultaneously. That requires multiple agents working together. Paperclip's job is to give those agents structure: a boss, a title, a job description, and a goal. Julian's metaphor is "company structure." The orchestration layer turns a pile of one-off agents into something that resembles a real org chart with reporting lines and a shared mission.

**Creating an agent in the Paperclip UI.** Navigate to the Agents tab. Click create-new-agent. The agent shows up running immediately. Julian labels his first one the CEO agent ("the CEO of our company"). Each agent slot exposes a description-of-what-it-can-do field, an optional prompt template, a title field, and an "adapter" setting. The adapter is what Julian calls the model-backend connection. His CEO agent is set to Claude directly, and he notes you can change the adapter to point at a different model. Exact phrasing: "this is a Claude AI agent working locally running for us 24/7." He explicitly mentions you can mix-and-match adapters across the same Paperclip company, with OpenClaw, Claude Code, Codex, and Hermes all wired in as agents simultaneously. For Hermes specifically he points at the Hermes GitHub repo as the source of the adapter setup instructions: "if you want to set up Hermes as an AI agent inside your Paperclip company, you just go to the GitHub." One of his existing OpenClaw agents in the demo is "a bit messy" and on an old setup, so he deletes it on camera. The takeaway is that adapter wiring is per-agent and any agent system that publishes a Paperclip-compatible adapter can be slotted in.

**Step 1: setting the company goal.** Right after install, Paperclip prompts you for a goal. Julian's example: "build the number one AI note-taking app, grow that." That becomes the company mission, and every agent reports to that goal. His real example: "help grow the Profit Boardroom my automation community." The goal is editable later. Critical claim: every task in the system is "goal-aligned," meaning each task traces back to the mission, which is what keeps agents from drifting.

**Step 2: hiring the team.** Pick AI workers by type. Julian lists CEO agent, CTO agent, engineer agents as starter archetypes, but says "any agent works as well." Mix-and-match across OpenClaw, Claude Code, Codex, Hermes is supported.

**Step 3: approving the plan.** Once goal and team are set, the CEO agent starts proposing actions in the Paperclip Inbox. The Inbox is the human-in-the-loop surface: the CEO sends questions and proposals. Julian's live demo: the inbox surfaces a recommendation to "create a marketing agent." He approves it, the CEO begins running the task. The CEO then asks a follow-up: "who should they report to?" Julian responds. The system records "board responded: wants marketing agent, I'll check it out and create it." The CEO then drafts the new agent's spec: icon, title, role, name, capabilities, and the adapter (Claude vs OpenClaw vs whatever). Julian approves, and the new Facebook Marketing Specialist appears in the team panel. From there he can assign tasks like "come up with some ad ideas for our community" and the agent starts working. He places another task: "add creatives" → "create issue." The inbox surfaces a completion message when done.

**Monitoring & budgets, what makes Paperclip "special."** Schedule: agents wake on a schedule, check their tasks, do the work, report back. They run 24/7 on the heartbeat schedule. Budgets per agent: each agent has a spend limit. When it hits the limit, it stops. This is the cost-control primitive. Goal alignment: every task traces back to the company mission. Agents know what to do AND why. Ticketing system: every conversation tracked, every decision explained, full audit log. If something goes wrong, you can trace exactly why. Human-in-the-loop authority: "You're the boss." You approve hires, override bad strategies, pause or remove agents. Full control. Phone monitoring: the whole dashboard is viewable from your phone, including run activity, status, issues. Julian frames the pre-Paperclip pain explicitly: "you had like 20 different Claude tabs open. You can't remember which tab does what. You sort of reboot, you lose everything, you copy and paste in context from like five different places to remind your agents what to do." Paperclip's organizational claims against that pain: tasks organized like tickets, conversations threaded, sessions survive reboots, context flows automatically, heartbeat-schedule tasks for recurring work.

**Paperclip vs Multica verdict.** Julian's stated preference: Paperclip is better. He likes both. Specific reasons for Paperclip: more control vs Multica, more customization, "things are actually working" whereas Multica had data-sync issues for him, more intuitive, "less buggy basically." He hedges that Multica is still a cool option and recommends checking both. He mentions he might do a future comparison video.

**Multica introduction.** "Today we're going to be testing out something called Multica that allows you to manage all your agents in one single platform. Similar to for example OpenClaw, but the difference is you can manage all of your agents as a team together." Headline pitch: "a zero human autonomous AI agent team that's working together." Free open-source project. Julian also notes a recent feature integration: Multica just absorbed the Routines feature that Claude Desktop introduced, slotted directly into the Multica UI.

**Multica's five big improvements/features.** First, agents as teammates: assign work to an agent the way you'd assign to a colleague. Agents have profiles, show up on a board, post comments, create issues, report blockers. Second, autonomous agents: set tasks and forget them. Full task lifecycle management with real-time progress. Third, reusable skills for agents (mentioned but not deep-dived in the chapter). Fourth, unified runtimes (mentioned, not deep-dived). Fifth, multi-workspace: organize work across the whole team across separate workspaces.

**Multica install path.** Two commands shown in sequence. The first command installs Multica (Mac OS or Linux). The second opens the login URL so you can authenticate. After auth and configuration, you can start setting up the agent team.

#### Chapter 12-13 — Setting up Multica + Hermes vs OpenClaw head-to-head intro

**Creating the first Multica workspace.** Julian creates a workspace called "Goldie Agency." After naming, the wizard immediately presents two agent type options: master agent (manages workspaces, science, tasks, coordinates work; the CEO of the agent team) or coding agent (checks code, implements features). Julian selects master agent.

**Runtime auto-detection (Hermes / OpenClaw / Claude).** Once master agent is selected, Multica automatically detects all locally running compatible runtimes. In Julian's case it picks up Hermes, OpenClaw, and Claude. He picks Hermes as the runtime for this CEO agent specifically because "that is my favorite agent at the minute." Auto-detection is one of the non-trivial Multica features: agents you have running locally (Claude, OpenClaw, Hermes, Open Code) all appear as selectable runtimes without manual configuration.

**Workspace vs Private visibility toggle.** Each agent has a visibility scope toggle with two values: private and workspace. Private means only the agent's creator can assign tasks to it. Workspace means all members can assign tasks. This becomes relevant for actual teams with multiple human members invited.

**Agent configuration surface (instructions, skills, tasks, environment, settings).** Instructions: free-form prompt where you tell the agent who it is. Equivalent of a system prompt at the agent level. Skills: individual capabilities. Local runtime skills are always available automatically and don't need manual addition. Tasks: to-do items. Julian's first goal: "saying hello to the team" as a smoke test. Environment: runtime environment configuration. Settings: operational settings including a maximum-concurrent-tasks counter, the runtime selector, and an online status indicator showing whether each runtime is currently reachable. Online status shows agents as either online or offline, and within online state shows them as either idle or working.

**Goal field, project creation, and the inbox protocol.** Each agent has a goal field. Julian sets the CEO Hermes agent's goal to "grow the AI Profit Boardroom." He creates the workspace's first project: "AI Profit Boardroom marketing strategy." Projects are containers for related tasks with their own member assignments and shareable links. The inbox is where agents post working updates and the human replies. After the first agent boots, it posts: "hey team, I'm a local coding agent running in multiple workspace. Here's a bit of information about me..." Julian replies "okay great what are you going to do next?" demonstrating the conversational protocol. The pattern is bidirectional: agents post status and questions, humans reply, the project state advances.

**Project metadata.** Each project has editable fields for description, status (planned, in progress, in review, done, blocked), priority (low, medium, high), and lead. The lead field can be assigned to either the human user or to any of the workspace's agents. Julian assigns the CEO Hermes agent as lead and switches status from planned to in progress with priority medium.

**Kanban board.** Multica's main board view is a Linear-style Kanban with backlog, to-do, in progress, in review, done, blocked. Issues flow through the columns as agents work them. Each task tile shows token usage for that task once an agent has worked on it. This is the key cost-visibility lever.

**Adding multiple agents.** Julian builds out the team: a marketing agent assigned to Claude Code runtime ("Your job is to come up with new ideas for marketing the AI Profit Boardroom. Can you work with the CEO and also come up with some ideas for adding new agents to the team?"); an OpenClaw agent ("make sure that you proactively come up with new automations that we use to save time in marketing"). Three agents now run in the same workspace: CEO Hermes (Hermes runtime), marketing agent (Claude Code runtime), OpenClaw agent (OpenClaw runtime). Julian emphasizes that any locally running agent can be added without restriction. The framework is runtime-agnostic.

**Persistent skill learning across the team.** Every time an agent solves a problem, the solution gets saved automatically as a skill. The next time the same problem comes up, the agent already knows how to handle it. Julian frames this as "your whole team gets smarter over time automatically." This is the same pattern as Hermes's self-improvement loop, but applied at the orchestration layer rather than per-agent. The skill registry is shared across the workspace.

**Single dashboard.** "You can see all your agents in one place. You see what they're working on. You see if something went wrong and there's no more switching between 10 different tools." Instead of N terminal windows for N agents, you open one Multica dashboard.

**Runtimes panel.** Multica has a dedicated runtimes view separate from the agents view. Shows for each agent: input, output, tokens used, where the agent is running (CLI, CLI 2, etc.), test connection button, metadata about creation time and history of work performed. Operational-observability surface for debugging which runtime is consuming the most tokens or whether a specific runtime instance is reachable.

**Workspace settings.** Members (invite real human collaborators), repositories (code repositories associated with the agent teams), general (delete workspace, add context, change URL slug, change name), APIs (edit external integrations), theme (light/dark/system mode).

**Routines (coming soon).** Julian flags routines as a feature he had referenced earlier but cannot demo because his version of Multica doesn't show it yet. Will eventually support scheduled or recurring agent task patterns.

**Multica vs Paperclip.** Multica is built for teams (multiple humans plus agents working together, member invites are first-class). Paperclip is built for solo users (one human running the show). Julian's recommendation: if you have a team, Multica is the better choice. If you're flying solo, either works. Both tools "are quite buggy as well sometimes" and operators should expect agents to occasionally fail or fail to connect properly.

**Hermes vs OpenClaw head-to-head opening.** "Who is the best agent? Which agent should I use? Is Hermes better than OpenClaw or is OpenClaw better than Hermes?" Julian has tested both for "probably hundreds of hours" and his personal favorite is Hermes. But: "the answer isn't what you think and I'll explain why that is in a minute." The actual head-to-head extends into Chapter 14.

#### Chapter 14-15 — Which agent wins, and the Hermes Dashboard v0.9 walkthrough

**Daily-driver strategy: run multiple agents concurrently.** Julian's stated workflow is to use Hermes and OpenClaw simultaneously every day, because both ship breaking changes constantly. He demonstrates live: he opens OpenClaw, it returns "config invalid" and refuses to load. He immediately switches to Hermes in terminal, which loads cleanly. The thesis is that any single open-source agent harness will be broken on any given day, so the only resilient pattern is to keep multiple harnesses installed and switch when one breaks. He explicitly says "every time there's a new release, a new version" something can break, and cites Hermes v0.9 having shipped today (the day of recording) as the trigger for the breakage he's about to show.

**The third agent in the rotation: Manis.** Julian classifies Manis as "underrated" specifically because it's not open-source. He frames it as the reliability backstop: "if everything else fails or breaks on me, I'm going with Manis." So the full stable is Hermes (primary), OpenClaw (secondary), Manis (always-on hosted fallback). He mentions Minimax as another hosted backup option in the same role.

**Cross-agent collaboration via Telegram group chats.** Julian shows two Hermes agents talking to each other inside a Telegram chat. He extends this to mixed-agent group chats: OpenClaw and Hermes in the same Telegram channel, talking together. The pattern many users adopt is Hermes-as-supervisor over OpenClaw, which "works really well." Group chats can mix two Hermes agents, multiple OpenClaw agents, Hermes plus OpenClaw side by side, or Hermes acting as supervisor for one or more OpenClaw workers.

**Workaround for the OpenClaw config-invalid break.** Julian fixes the broken OpenClaw by switching the model behind it to GLM 5.1. He runs "GLM 5.1 with OpenClaw just to fix the original setup." Implicit lesson: when OpenClaw breaks on a config error, swapping the model backend can recover the harness. He frames the silver lining: "you become better and better at using terminal" by running these tools.

**Hermes Dashboard v0.9 sections.** Status (agent health), Sessions (manage running sessions across different agents/models), Analytics (usage metrics), Logs (operational log surface), Scheduled tasks / cron jobs (list with start, trigger, and delete actions per task), Skills (with category switching, visibly cleaner than OpenClaw's equivalent). Managing multiple agents is non-trivial because each agent has its own model, session, chat, context, memory plugin, and skill set. Without a UI dashboard you're tracking all that across terminal windows, Telegram threads, and config files. The dashboard collapses that into one navigation surface.

**The biggest gap: no in-dashboard chat (yet).** You cannot chat directly with Hermes inside the dashboard. To talk to a Hermes agent you still need terminal, Telegram, or another messaging platform. Julian predicts "Hermes will definitely have to add a chat inside the dashboard at some point" but it doesn't exist in v0.9. This is OpenClaw's one clear win in the head-to-head.

**Side-by-side cron section comparison.** OpenClaw's layout is "quite difficult to understand," "quite messy," "look at the way this is laid out and everything else." Hermes' equivalent is "much cleaner, easier to understand, looks nicer in general, an easier UI to navigate." Same verdict on the skills section. Hermes adds category switching inside skills, letting you filter by category, which OpenClaw's skills view does not.

**OpenRouter usage trends.** OpenClaw "usage is actually dropping quite a lot on OpenClaw directly right now." Currently ranked number one on OpenRouter but trending down. Hermes "look at the growth of that project. That's a trajectory where you can see people are using it more and more." Currently ranked number two but on a clear upward curve. He calls them "neck and neck in terms of usage" right now, but treats the trend differential as a signal that users are migrating from OpenClaw to Hermes. He attributes the OpenClaw decline to community complaints surfacing across forums: doesn't work well, struggles, breaks all the time, updates are temperamental. He specifically notes that even GPT (which he says "Peter Steinberger actually moved over and partnered with") doesn't configure properly with OpenClaw, "quite sad."

**Noose Research vs Peter Steinberger origin stories.** Hermes / Noose Research: "one of the biggest AI labs in the world." Builds their own foundation models, not just agent harnesses. Hermes itself is also a model name available from the Noose Research portal inside Hermes the harness (so the same brand spans model + control center). Engineered "properly to work as well as it possibly can" because they understand both layers. OpenClaw / Peter Steinberger: "has kind of been like a DIY project where Peter Steinberger had just built it over a weekend and then it's grown from there." Has an "amazing community" but "wasn't built with the foundations of an amazing AI lab." "Wasn't built from the ground up as something that was designed to be used worldwide and become the number one leading AI agent in the world."

**OpenClaw advantages Julian still concedes.** Bigger community ("OpenClaw for sure has a much bigger community"). More skills (larger contributor pool). More support. More ecosystem projects. In-dashboard chat (the one direct UX win).

**Skill compounding argument: learn the discipline, not the tool.** Julian addresses "which one should I learn" with a meta-argument: it doesn't matter because the skill set compounds. Building agents in either Hermes or OpenClaw teaches you the discipline of agent setup and use-case automation. When the next-generation tool ships (which he says is inevitable, citing that OpenClaw only launched in November 2025 and Hermes only launched in February 2026), users who already understand agent harnesses will be in an "advanced situation." #### Chapter 44 — Fast Mode Explained, How priority routing changes everything

**What Fast Mode is and how to activate it.** Fast Mode is invoked by typing `/f fast` inside Hermes. Once activated, the user's requests jump to the front of the inference line. Julian describes this as "priority processing cues" (queues), "much lower latency," and "instant responses." This is not a different model and not a faster sampling setting; it is a routing change at the inference-queue layer. The same model the user was already running will respond on a priority lane instead of the default lane.

**Where the optimization lands.** Fast Mode is targeted at GPT 5.4 and Claude Fast Tier specifically. Julian states: "it's optimized specifically for GPT 5.4 and Claude fastier" (Fast Tier). On those two models the per-step latency drops materially. "Coding loops that used to take 5 seconds now take one." That is a 5x improvement on a single agent decision step. The compounding effect is what matters because agents are loop-based: an agent that takes 5 steps to complete a task at 5 seconds per step takes 25 seconds total in standard mode and 5 seconds in Fast Mode.

**Why priority routing exists in the first place.** Prior to Fast Mode, agents had a structural pacing problem. The user asks for something, the agent thinks, pauses, executes one tool call, pauses again, executes the next, and so on. Clunky, "talking to someone with a two-second delay on every response." The problem is not the model's reasoning quality; it is the inter-step latency, which compounds across multi-step autonomous workflows. Fast Mode addresses the queue depth between an agent's tool-call return and its next decision token. Compared to "the same kind of tech that pro-tier services use, speed lanes for your requests."

**Behavioral threshold thesis.** Speed changes user behavior. When an agent is slow, users avoid it, work around it, do tasks manually because waiting is more expensive than doing. When an agent is instant, users use it for everything, trust it, rely on it, and give it harder tasks. Fast Mode makes Hermes "cross that threshold" from "tool you use sometimes" to "tool you use constantly." The unlock is not the seconds saved; it is the change in what the user is willing to delegate. "When something responds instantly, you trust it more. You give it harder tasks. You let it do more. That's the real unlock."

#### Chapter 51-52 — Hermes + Karpathy LLM Wiki + Full Hermes Feature Overview

**The problem Karpathy's LLM wiki solves.** Most people use AI documents wrongly: upload files, ask questions, lose everything when the session ends. Every new question forces the model to rediscover the same information from raw sources. The standard RAG pattern: works, but no memory, no accumulation, no compounding. Andre Karpathy, founding team member of OpenAI and former director of AI at Tesla, published a document on April 4th, 2026 that earned over 5,000 stars on GitHub in under 48 hours describing an alternative pattern called LLM wiki. Instead of retrieving from raw documents on every question, the AI builds and maintains a persistent wiki: a structured, interlinked collection of markdown files that sits between the user and the raw sources. New sources are read, key information is extracted, existing pages are updated, summaries are revised, and contradictions between new and old claims are flagged.

**Karpathy's framing.** The LLM is the programmer, the wiki is the codebase, Obsidian is the viewer on the other side. The wiki is a persistent compounding artifact. Cross references are already there. Contradictions are already flagged. It keeps getting richer with every source added.

**Hermes packages LLM wiki as a built-in skill.** Run `Hermes update` to pull the latest version, then use the slash command `/llm wiki` followed by your research topic. Hermes starts building the wiki. Noose Research demonstrated this on their own projects, web sources, code, and research papers; Hermes read through all of it and produced a structured knowledge base across everything Noose Research has been building. "Weeks of manual effort" condensed.

**Three layers of the LLM wiki architecture.** Layer one is raw sources: documents, articles, papers, data files. These are immutable. The AI reads from them but never changes them. They are the source of truth. Layer two is the wiki itself: a folder of AI-generated markdown files including summaries, entity pages, concept pages, comparisons, and synthesis. The AI owns this layer. It creates pages, updates them when new sources arrive, maintains cross references, and keeps everything consistent. The user reads it; the AI writes it. Layer three is the schema: a configuration document that tells the AI how the wiki is structured, what the conventions are, and what to do when new sources are added or questions are asked. The schema is what makes the AI a disciplined knowledge organizer rather than a generic chatbot. The user and the agent evolve the schema together over time.

**Three core operations: ingest, query, lint.** Ingest: drop a new source in and tell the AI to process it. It reads the document, pulls out key takeaways, writes a summary page, updates the index, and touches any relevant pages across the wiki. A single source might update 10 to 15 different pages. That is where compounding starts. Query: ask a question. The AI searches relevant pages and gives a synthesized answer with citations. Karpathy's underrated insight: good answers can be filed back into the wiki as new pages. Lint: ask the AI to health-check the wiki. Look for contradictions, stale information, orphan pages, missing connections. Lint also suggests new questions to investigate and new sources to find.

**Who this is for.** Researchers going deep on a topic over weeks or months. Content creators researching for videos or articles. Developers (Hermes can read the codebase, docs, and papers and keep everything organized). Business teams (feed Slack threads, meeting transcripts, and customer calls into a wiki the AI keeps current).

**Karpathy's insight on maintenance burden.** The tedious part of maintaining a knowledge base is not the reading or thinking, it is the bookkeeping, updating cross references, keeping summaries current. Humans abandon wikis because the maintenance burden grows faster than the value. An AI agent does not get bored, does not forget a cross reference, and can touch 15 files in one pass.

**Practical setup tips.** Install Hermes Agent with one command. Run `Hermes update` to get the latest version with the LLM wiki skill included. Type `/llm wiki <topic>` to start a wiki. Use Obsidian as the viewer. Use the Obsidian Web Clipper browser extension to grab articles as markdown, drop them into the raw sources folder, and tell the agent to ingest. Two navigation files inside the wiki: the index is a catalog of everything with one-line summaries per page, the log is a chronological record of what the agent has done. The agent reads the index first before answering any question. The schema is worth investing time in.

**Karpathy's historical framing: Vannevar Bush's Memex.** Karpathy ties LLM wiki back to Vannevar Bush's Memex from 1945: a personal knowledge store with associative trails between documents, private, actively curated. The piece Bush could not solve was who does the maintenance. Now there is an answer.
